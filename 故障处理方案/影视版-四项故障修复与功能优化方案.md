# 影视版 — 四项故障修复与功能优化方案

## 一、故障1：作品管理点击图片添加为参考图后自动切换标签页

### 故障现象
在"图片生成"页面的左侧"作品管理"标签页点击图片添加为参考图片后，会自动切换到"参考图片"标签页。

### 故障定位
**文件**: `lib/widgets/reference_sidebar.dart`
**代码位置**: `_addImageToReference()` 方法（约第55-68行）

```dart
Future<void> _addImageToReference(String path) async {
    final file = File(path);
    if (await file.exists()) {
      // ... 添加图片逻辑 ...
      
      // ★ 故障根因：这行代码强制切换到第一个标签页（参考图片）
      _tabController.animateTo(0);
    }
  }
```

### 故障根因
`_addImageToReference()` 方法末尾调用了 `_tabController.animateTo(0)`，每次添加参考图后都会强制切换到"参考图片"标签页（index=0）。

### 修复方案
**删除** `_tabController.animateTo(0)` 这行代码。用户在"作品管理"标签页点击图片时，只需将图片添加到参考图列表，不应自动切换标签页。可以改为显示一个轻量提示（SnackBar），告知用户图片已添加成功。

```dart
Future<void> _addImageToReference(String path) async {
    final file = File(path);
    if (await file.exists()) {
      final bytes = await file.readAsBytes();
      final fileName = file.path.split(Platform.pathSeparator).last;

      if (ref.read(uploadedImagesProvider).any((img) => img.name == fileName)) {
        return;
      }

      final image = UploadedImage(
        id: const Uuid().v4(),
        name: fileName,
        path: path,
        base64: base64Encode(bytes),
        bytes: bytes,
      );
      ref.read(uploadedImagesProvider.notifier).addImage(image);
      ref.read(selectedImagesProvider.notifier).addImage(image);

      // ★ 修复：移除自动切换，改为提示
      if (mounted) {
        ScaffoldMessenger.of(context).showSnackBar(
          SnackBar(content: Text('已添加参考图: $fileName'), duration: const Duration(seconds: 1)),
        );
      }
    }
  }
```

### 影响范围
- 仅影响 `reference_sidebar.dart` 一个文件
- 不影响其他功能

## 二、故障2：重绘编辑器裁切后发送的是完整图片

### 故障现象
"重绘编辑器"里面的"裁切"标签页，裁切图片后发送到聊天窗口的图片还是完整图片，不是裁切后的图片。

### 故障定位
**文件**: `lib/screens/home_screen.dart`（第72-96行）和 `lib/widgets/reference_sidebar.dart`

**调用链分析**:
1. 重绘编辑器的"提交生成"按钮调用 `widget.onGenerate(prompt, widget.imagePath, croppedBytes: croppedBytes)`
2. `onGenerate` 回调定义在 `home_screen.dart` 的 `onRepaintGenerate` 参数中
3. 关键代码：

```dart
onRepaintGenerate: (prompt, imagePath, {Uint8List? croppedBytes}) async {
    // ...
    String base64Str;
    String refPath = imagePath;  // ★ 问题1：refPath 始终是原始图片路径
    if (croppedBytes != null) {
      base64Str = base64Encode(croppedBytes);  // base64 用的是裁切后的数据 ✓
    } else {
      final file = File(imagePath);
      if (!await file.exists()) return;
      final bytes = await file.readAsBytes();
      base64Str = base64Encode(bytes);
    }
    final params = ref.read(generateParamsProvider);
    ref.read(generateLogicServiceProvider).generate(
      prompt: prompt,
      model: params.model,
      aspectRatio: params.aspectRatio,
      imageSize: params.imageSize,
      referenceImages: [base64Str],       // ★ 这里传的是裁切后的base64 ✓
      referenceImagePaths: [refPath],     // ★ 问题2：路径还是原图路径
    );
},
```

### 故障根因
有两个层面的问题：

1. **API发送层面**：`referenceImages` 传的 base64 确实是裁切后的数据，所以 API 生成时用的参考图是正确的。
2. **聊天窗口显示层面**：`referenceImagePaths` 传的是原始图片路径 `imagePath`（即 `widget.imagePath`），聊天窗口的消息气泡显示图片时用的是这个路径，所以显示的是完整原图。

核心问题：当有裁切数据时，需要先将裁切后的图片保存为临时文件，然后用新路径作为 `referenceImagePaths`。

### 修复方案
在 `home_screen.dart` 的 `onRepaintGenerate` 回调中，当 `croppedBytes` 不为空时，先将裁切后的图片保存到 `data/input/` 目录，然后用新路径替换 `refPath`：

```dart
onRepaintGenerate: (prompt, imagePath, {Uint8List? croppedBytes}) async {
    final configs = ref.read(apiConfigsProvider);
    if (configs.isEmpty) {
      if (mounted) {
        ScaffoldMessenger.of(context).showSnackBar(
          const SnackBar(content: Text('请先配置API')),
        );
      }
      return;
    }
    String base64Str;
    String refPath = imagePath;
    if (croppedBytes != null) {
      base64Str = base64Encode(croppedBytes);
      // ★ 修复：将裁切后的图片保存为临时文件
      try {
        final exePath = Platform.resolvedExecutable;
        final appDir = File(exePath).parent.path;
        final inputDir = Directory('$appDir/data/input');
        await inputDir.create(recursive: true);
        final timestamp = DateTime.now().millisecondsSinceEpoch;
        final croppedPath = '${inputDir.path}/cropped_$timestamp.png';
        await File(croppedPath).writeAsBytes(croppedBytes);
        refPath = croppedPath;  // 用裁切后的路径
      } catch (e) {
        debugPrint('保存裁切图片失败: $e');
      }
    } else {
      final file = File(imagePath);
      if (!await file.exists()) return;
      final bytes = await file.readAsBytes();
      base64Str = base64Encode(bytes);
    }
    final params = ref.read(generateParamsProvider);
    ref.read(generateLogicServiceProvider).generate(
      prompt: prompt,
      model: params.model,
      aspectRatio: params.aspectRatio,
      imageSize: params.imageSize,
      referenceImages: [base64Str],
      referenceImagePaths: [refPath],  // ★ 现在用的是裁切后的路径
    );
},
```

### 影响范围
- 主要修改 `home_screen.dart` 的 `onRepaintGenerate` 回调
- 会在 `data/input/` 目录下生成裁切临时文件（以 `cropped_` 前缀命名）

## 三、故障3：影视资产库后台生成无法正确反馈进度和添加到资产库

### 故障现象
"影视资产库"添加资产时，点击后台生成后，无法正确的反馈进度和添加到资产库，但在"作品管理"页面已经看到了生成返回的资产。

### 故障定位
**文件**: `lib/screens/asset_library_screen.dart`
**方法**: `_runBackgroundGenerate()`（约第999-1070行）

### 故障根因
分析 `_runBackgroundGenerate` 方法发现两个关键问题：

**问题1：后台生成完成后没有保存资产到资产库**
后台生成模式下，对话框已经关闭（`Navigator.pop`），`_runBackgroundGenerate` 方法只是：
- 下载图片到 `assets/{category}/` 目录
- 更新 `extraImages` 列表（但这个列表是对话框的局部变量，对话框已关闭，更新无意义）
- 更新 `backgroundTaskProvider` 的进度

但**没有调用 `assetProvider.notifier` 来保存/更新资产到资产库的持久化存储**。图片虽然下载到了 output 目录（所以作品管理能看到），但资产库的 JSON 数据没有更新。

**问题2：`extraImages` 是对话框的局部变量**
后台模式下对话框已关闭，`extraImages` 的引用虽然还在闭包中，但修改它不会触发任何 UI 更新，也不会持久化。

### 修复方案
在 `_runBackgroundGenerate` 方法完成后，需要：
1. 构建完整的 `Asset` 对象
2. 调用 `assetProvider.notifier` 将资产保存到持久化存储
3. 刷新资产列表

需要将资产的基本信息（名称、类别、描述、主图路径）也传入 `_runBackgroundGenerate`，生成完成后自动创建/更新资产：

```dart
Future<void> _runBackgroundGenerate(String taskId, List<Map<String, String>> views,
    String mainImagePath, String assetName, String category, Map<String, String> params,
    List<AssetRefImage> extraImages, StateSetter setDialogState, int initialImageCount,
    {String description = ''}) async {  // ★ 新增 description 参数
    
    final apiService = ref.read(apiServiceProvider);
    final assetService = ref.read(assetServiceProvider);

    // ... 现有的生成逻辑保持不变 ...

    await Future.wait(futures);

    ref.read(backgroundTaskProvider.notifier).updateTask(taskId,
      description: failed > 0 ? '完成 (${views.length - failed}成功, $failed失败)' : '全部完成',
      progress: 1.0,
      isComplete: true,
    );

    // ★ 修复：生成完成后，自动创建/保存资产到资产库
    final validImages = extraImages.where((e) => e.path.isNotEmpty).toList();
    if (validImages.isNotEmpty || mainImagePath.isNotEmpty) {
      final asset = Asset(
        id: const Uuid().v4(),
        name: assetName,
        category: category,
        description: description,
        imagePath: mainImagePath,
        images: extraImages,
        createdAt: DateTime.now(),
      );
      await ref.read(assetProvider.notifier).addAsset(asset);
      // 刷新资产列表
      await ref.read(assetProvider.notifier).loadAssets();
    }
}
```

同时需要修改 `_generateViews` 中调用 `_runBackgroundGenerate` 的地方，传入 `description` 参数。

### 影响范围
- 修改 `asset_library_screen.dart` 的 `_runBackgroundGenerate` 方法
- 修改 `_generateViews` 方法中的调用参数
- 可能需要检查 `AssetProvider` 的 `addAsset` 方法是否支持持久化

## 四、功能4：后台任务保活气泡按钮

### 需求描述
切换页面后，影视工坊的拆解分镜进程、通读全文功能被打断且无法恢复。需要在左侧"关于绘云AI"下面增加一个气泡按钮保活后台进程，点击气泡后展示正在处理的进程，点击对应进程可以自由切回对应界面。

### 故障根因分析
**页面切换导致 widget 被销毁的机制**：

`home_screen.dart` 的 `_buildCurrentScreen()` 方法使用 `switch` 直接返回新的 widget 实例：
```dart
Widget _buildCurrentScreen() {
    switch (_selectedIndex) {
      case 0: return const FilmWorkshopWrapper();
      case 1: return GenerateScreen(promptController: _promptController);
      // ...
    }
}
```

每次切换页面时，之前的 widget 会被销毁（dispose），新的 widget 会被创建。`FilmWorkshopScreen` 中的 `_splitScriptAndDisplay` 和 `_analyzeScript` 方法是在 widget 的 State 中执行的异步操作，widget 被销毁后：
- `mounted` 变为 false，后续的 `setState` 和 UI 更新不再执行
- 但 `await` 的 Future 本身不会被取消（Dart 的 Future 不支持取消），API 请求可能还在进行
- 流式响应（Stream）的订阅会因为 widget dispose 而丢失

### 设计方案

#### 4.1 架构改造：将长时间任务从 Widget 层迁移到 Provider 层

**核心思路**：创建一个全局的 `BackgroundProcessProvider`，管理所有长时间运行的任务。任务不再绑定在 widget 生命周期上，而是由 Provider 持有，页面切换不会影响任务执行。

**新增文件**: `lib/providers/background_process_provider.dart`

```dart
/// 后台进程模型
class BackgroundProcess {
  final String id;
  final String title;
  final String description;
  final double progress;       // 0.0 ~ 1.0
  final bool isComplete;
  final bool isFailed;
  final String? error;
  final int targetPageIndex;   // ★ 关联的页面索引，点击可跳转
  final String processType;    // 'split_script' | 'analyze_script' | 'asset_generate' | 'batch_generate'
  final Map<String, dynamic>? metadata;  // 存储任务相关的额外数据

  BackgroundProcess({...});
}

class BackgroundProcessNotifier extends StateNotifier<List<BackgroundProcess>> {
  BackgroundProcessNotifier() : super([]);

  void addProcess(BackgroundProcess process) { ... }
  void updateProcess(String id, {...}) { ... }
  void removeProcess(String id) { ... }
  void clearCompleted() { ... }
  
  /// 获取活跃进程数量
  int get activeCount => state.where((p) => !p.isComplete && !p.isFailed).length;
}
```

#### 4.2 导航侧边栏气泡按钮

**修改文件**: `lib/widgets/navigation_sidebar.dart`

在"关于绘云AI"导航项下方增加一个气泡按钮，当有活跃后台进程时显示：

```
  ⚙️ 设置
  ℹ️ 关于绘云AI
  ─────────────
  🔵 3 个任务运行中  ← 气泡按钮（有任务时显示，无任务时隐藏）
```

**气泡按钮设计**：
- 圆角矩形，带脉冲动画（呼吸灯效果），表示有任务在运行
- 显示活跃任务数量
- 点击后弹出浮层面板，展示所有进程列表

点击气泡按钮后，在侧边栏右侧弹出一个浮层面板：


**交互逻辑**：
- 每个进程条目可点击，点击后自动切换到对应页面（通过 `targetPageIndex`）
- 已完成的进程显示绿色勾号，失败的显示红色叉号
- 进行中的进程显示进度条和百分比

#### 4.4 页面保活改造

**修改文件**: `lib/screens/home_screen.dart`

将 `_buildCurrentScreen()` 改为使用 `IndexedStack`，这样所有页面都会保持在内存中，切换页面不会销毁 widget：

```dart
// 修改前：
Widget _buildCurrentScreen() {
    switch (_selectedIndex) {
      case 0: return const FilmWorkshopWrapper();
      case 1: return GenerateScreen(promptController: _promptController);
      // ...
    }
}

// 修改后：
Widget _buildCurrentScreen() {
    return IndexedStack(
      index: _selectedIndex,
      children: [
        const FilmWorkshopWrapper(),
        GenerateScreen(promptController: _promptController),
        const AssetLibraryScreen(),
        const StoryboardGalleryScreen(),
        const GalleryScreen(),
        const ApiConfigScreen(),
        const SettingsScreen(),
        const AboutScreen(),
      ],
    );
}
```

**注意**：`IndexedStack` 会让所有页面同时存在于内存中，内存占用会增加。但对于桌面应用来说这是可接受的，而且能彻底解决页面切换中断任务的问题。

#### 4.5 任务迁移到 Provider 层（长期方案）

对于拆解分镜和通读全文这类核心任务，应该将执行逻辑从 `FilmWorkshopScreen` 的 State 方法中迁移到独立的 Service + Provider 中：

**新增/修改文件**:
- `lib/services/script_analysis_service.dart` — 通读全文的业务逻辑
- `lib/services/storyboard_service.dart` — 拆解分镜的业务逻辑（已有，需增加后台执行支持）
- `lib/providers/background_process_provider.dart` — 统一管理后台进程

这样即使不用 `IndexedStack`，任务也不会因为页面切换而中断。
